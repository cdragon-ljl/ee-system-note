# 嵌入式知识体系 · #15 · NVIC 中断优先级：抢占优先级与子优先级的博弈

---

> 你的程序有两个中断：USART1 收数据，TIM2 做定时。
>
> USART1 进来时 TIM2 刚好也来了——谁先执行？
>
> USART1 执行到一半，TIM2 又触发了——会不会打断？
>
> TIM2 正在跑，USART1 来了——能不能抢？

这些问题，答案全在 **NVIC（Nested Vectored Interrupt Controller）** 的优先级系统里。搞不懂它，多中断协同就是个黑盒子。

---

## 一、优先级分组：抢占 vs 子优先级

Cortex-M 内核的 NVIC 支持最多 **256 级中断优先级**（实际芯片会裁剪，STM32F4 只用 16 级），通过 `PRIGROUP` 寄存器分成两组：

- **抢占优先级（Preempt Priority）**：决定是否**打断**正在执行的中断
- **子优先级（Sub Priority）**：决定同时到达时**谁先排队**

用 4 位表示优先级时，PRIGROUP 的配置决定了这两个部分各占多少位：

| PRIGROUP | 抢占优先级位数 | 子优先级位数 | 抢占级数 | 子优先级级数 |
|:-------:|:------------:|:-----------:|:-------:|:----------:|
| 0 (默认) | 4 | 0 | 16 | 1 |
| 1 | 3 | 1 | 8 | 2 |
| 2 | 2 | 2 | 4 | 4 |
| 3 | 1 | 3 | 2 | 8 |
| 4 | 0 | 4 | 1 | 16 |

### 关键规则

1. **抢占优先级高的中断可以打断抢占优先级低的中断**
2. **抢占优先级相同的中断不会互相打断**
3. **抢占优先级相同、子优先级不同的中断同时到来时，子优先级高的先执行**
4. **抢占优先级相同的中断正在执行时，同优先级的新中断会挂起等待，不会嵌套**

### 实际使用建议

大多数嵌入式应用使用 **抢占 4 位 + 子优先 0 位**（PRIGROUP=0），也就是 16 个独立优先级。如果需要精细控制同一优先级的中断排队顺序，才考虑分组。

> 新手最容易犯的错误：设了不同的子优先级但抢占优先级相同，然后期望一个中断打断另一个。**两个中断抢占优先级相同，永远不能互相嵌套。**

## 二、NVIC_SetPriority() 的正确用法

HAL 库把优先级包装成两个参数：

```c
HAL_NVIC_SetPriority(IRQn_Type IRQn, uint32_t PreemptPriority, uint32_t SubPriority);
```

### 例子：正确配置

假设 PRIGROUP = 2（2 位抢占 + 2 位子优先级，共 4 级抢占 × 4 级子优先）：

```c
HAL_NVIC_SetPriority(TIM2_IRQn, 1, 0);    /* 抢占级 1，子优先 0 */
HAL_NVIC_SetPriority(USART1_IRQn, 0, 3);  /* 抢占级 0，子优先 3 */
HAL_NVIC_SetPriority(DMA2_Stream0_IRQn, 0, 1);  /* 抢占级 0，子优先 1 */
HAL_NVIC_EnableIRQ(TIM2_IRQn);
HAL_NVIC_EnableIRQ(USART1_IRQn);
HAL_NVIC_EnableIRQ(DMA2_Stream0_IRQn);
```

这套配置的含义：

- USART1 和 DMA2 **抢占优先级同为 0**，不会互相打断。同时到达时，DMA2（子优先=1）比 USART1（子优先=3）先执行
- TIM2（抢占=1）可以被 USART1 或 DMA2 **打断**
- TIM2 正在执行时，USART1 和 DMA2 不会理它——因为 TIM2 的抢占级更低

### 坑：NVIC 优先级数值方向

**经常有人搞反**：数值越小，优先级越高。

- 抢占优先级 `0` > `1` > `2` > `3`
- 子优先级同理：`0` > `1` > `2` > `3`

### 手动寄存器操作

如果你不用 HAL 库：

```c
/* 设 USART1 抢占优先级为 0，子优先级为 3 */
/* 在 2+2 分组下，优先级寄存器的值为：0<<2 | 3 = 3 */
NVIC->IP[USART1_IRQn] = (0 << 4) | (3 << 0);  /* 实际值 = 0x03 */
```

`NVIC->IP[irq]` 寄存器占用 8 位，但只有高 4 位有效（STM32 只用 4 位优先级）。所以实际代码中通常写：

```c
NVIC_SetPriority(USART1_IRQn, (0 << 2) | 3);  /* (抢占<<PRIGROUP_BITS) | 子优先 */
```

## 三、中断嵌套的完整时序

来看一个具体的场景，直观展示抢占优先级的作用：

```
USART1 (抢占=0) ──[1]─────[4]───→
                │       │
TIM2  (抢占=1)  ──[2]───│──────→
                        │
主循环           ───────[3]─────────→
时间轴           →→→→→→→→→→→→→→→→→→
```

**时序说明：**

1. **①** USART1 中断触发，抢占优先级 0 → CPU 暂停主循环，进入 USART1 ISR
2. **②** TIM2 中断触发，抢占优先级 1 → 但因为 1 > 0，TIM2 **不能打断** USART1 → 挂起等待
3. **③** USART1 ISR 执行完毕 → CPU 检查挂起队列 → TIM2（抢占=1）进入执行
4. **④** 又在 TIM2 执行中触发了 USART1 中断 → USART1（抢占=0 < 1）**可以打断** TIM2 → 嵌套！
5. USART1 ISR 执行完毕 → 返回 TIM2 → TIM2 执行完毕 → 返回主循环

### 嵌套层数的上限

理论上 Cortex-M 支持**无限嵌套**，但受限于栈空间。每个嵌套的中断 ISR 至少消耗几十字节栈，实际项目中建议嵌套不超过 **3~4 层**。

## 四、中断延迟的测量与优化

**中断延迟（Interrupt Latency）** 是指从硬件触发中断到 ISR 第一条指令执行之间的时间。

### 哪些因素影响延迟

| 因素 | 典型耗时（Cortex-M4 @168 MHz） |
|:----|:------------------------------:|
| CPU 完成当前指令 | ~0 （指令级流水线） |
| 压栈（8 个寄存器） | ~12 个周期 |
| 取中断向量 | ~3 个周期 |
| 跳转 ISR | ~3 个周期 |
| **总延迟** | **~12~18 个周期 ≈ 70~110 ns** |

这是理想情况。实际中延迟会因以下情况显著增大：

### 1. 同优先级屏障

同一个优先级的 ISR 正在执行时，另一个同优先级的 ISR 即使已挂起也要等前面的跑完。**这是最常见的延迟来源**。

### 2. 关中断临界区

代码中调用了 `__disable_irq()` 或 `taskENTER_CRITICAL()`，中断被整体屏蔽。临界区多长，延迟就多长。

### 3. 高优先级 ISR 太长

高优先级的 ISR 如果写得太长（比如在里面做 `HAL_UART_Receive()` 轮询），后面所有低优先级 ISR 都会被堵住。

### 优化建议

**原则：ISR 只做最必要的事，数据处理交给任务层。**

```c
/* ❌ 坏做法：在 ISR 里做大量处理 */
void USART1_IRQHandler(void) {
    uint8_t data = USART1->DR;
    ring_buffer_write(&buf, data);     /* ✅ 到此为止 */
    /* 不要在这里做协议解析、写入 Flash、I2C 通信！ */
}

/* ✅ 好做法：标记 → 退出 ISR → 在主循环或任务中处理 */
void USART1_IRQHandler(void) {
    uint8_t data = USART1->DR;
    ring_buffer_write(&buf, data);
    rx_ready = 1;                      /* 设置标志位 */
    /* 或 HAL 方式：HAL_UART_RxCpltCallback() */
}
```

### 4. 中断优先级与系统节拍

如果用 FreeRTOS，注意 **SysTick 中断优先级**必须设置成**所有中断中最低**（或至少低于临界区屏蔽的级别），否则 `portDISABLE_INTERRUPTS()` 会关不掉 SysTick，导致系统崩溃。

FreeRTOS 的推荐配置：

```c
/* 在 FreeRTOSConfig.h 中 */
#define configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY   5  /* 可安全调用的最高优先级 */
#define configKERNEL_INTERRUPT_PRIORITY                15 /* SysTick 用最低优先级 */
```

## 五、实战：多中断优先级设计

一个典型的 STM32 产品中断优先级分配：

| 中断源 | 抢占优先级 | 子优先级 | 理由 |
|:------|:---------:|:-------:|:-----|
| HardFault | -1（固定） | - | 不可屏蔽，最高优先 |
| 紧急停机（GPIO EXTI） | 0 | 0 | 最快响应 |
| USB OTG | 0 | 1 | USB 有帧同步要求 |
| DMA 传输完成 | 1 | 0 | 数据流及时搬走 |
| UART 接收 | 1 | 1 | 收数据，别丢 |
| TIM 定时 | 2 | 0 | 允许稍等 |
| SysTick | 15 | 0 | RTOS 系统心跳 |

## 总结

NVIC 优先级系统说到底就三句话：

1. **抢占优先级**决定能不能打断，**子优先级**决定谁先排队
2. **数值越小，优先级越高**——记住这一条能避免 90% 的配置错误
3. **ISR 越短越好**——高优先级 ISR 里写长代码，等于在关别人的中断

搞懂了这三条，多中断协同就不再是玄学，而是一个可预测、可设计的系统。
